
# Embedding 模型上下文窗口详解

---

## 一、什么是上下文窗口？

### 1.1 基本概念

**上下文窗口（Context Window）** 是指 AI 模型在一次处理中能够接受的最大输入文本长度。

这个概念适用于两种主要的 AI 模型：

| 模型类型 | 上下文窗口含义 | 典型应用场景 |
|---------|--------------|-------------|
| **LLM（大语言模型）** | 能"记住"的对话历史长度 | 多轮对话、长文本生成 |
| **Embedding 模型** | 一次能向量化的最大文本片段长度 | RAG 文档分块、语义检索 |

### 1.2 核心区别

```
LLM 的上下文窗口：我能同时看多少历史对话
Embedding 的上下文窗口：我能一口吞下多长的文本片段
```

---

## 二、Embedding 模型的上下文窗口

### 2.1 定义

对于 Embedding 模型，上下文窗口是指：**模型能够一次性处理并转换为向量的最大文本长度（以 token 为单位）**。

### 2.2 为什么重要？

1. **分块的上限依据**
   - RAG 系统中，文档分块大小不能超过 Embedding 模型的上下文窗口
   - 超过窗口的文本会被截断，导致语义丢失

2. **语义完整性**
   - 文本过长 → 语义被稀释 → 向量质量下降
   - 文本过短 → 上下文信息不足 → 检索精度降低

3. **成本与性能平衡**
   - 窗口越大 → 模型越复杂 → 调用成本越高
   - 窗口越小 → 需要分更多块 → 检索成本增加

## 三、Token 是什么？

### 3.1 Token 的定义

**Token** 是 AI 模型处理文本的基本单位，相当于"词片段"。

### 3.2 Token 与字符的换算

| 语言 | 1 Token ≈ 字符数 |
|------|-----------------|
| 中文 | 1.5 - 2 字符 |
| 英文 | 4 - 5 字符（约 0.75 个单词）|
| 代码 | 3 - 4 字符 |

**示例：**
- "你好世界" → 约 2-3 tokens
- "Hello world" → 约 2 tokens

## 五、在 RAG 系统中的实际应用

### 5.1 分块大小的设计原则

```
分块大小 ≤ 上下文窗口 × 安全系数（0.7 - 0.8）
```

**原因：**
- 留出余量，避免文本意外超长被截断
- 考虑特殊字符、标点符号等额外 token 消耗

### 5.2 你的项目中的实践

根据项目代码分析：

```java
// 项目中使用的分块大小（约 500 中文字符）
// 对应 DashScope text-embedding-v2 的 2048 token 窗口
// 500 中文字符 ≈ 250-350 tokens，远小于 2048 的上限
```

**设计思路：**
1. **DashScope 窗口**：2048 tokens ≈ 1500-2000 中文字符
2. **安全系数**：0.25 - 0.33（非常保守）
3. **实际分块**：500 字符
4. **优势**：确保语义完整性，检索精度高

### 5.3 如果窗口太小怎么办？

**场景：** 你的文档有 3000 字符，但 Embedding 窗口只有 2000 字符

**解决方案：**
1. **分块策略**：将文档切分成多个小于 2000 字符的片段
2. **重叠分块**：相邻 chunk 有 50-100 字符重叠，避免语义断裂
3. **升级模型**：换用更大窗口的 Embedding 模型

## 六、与 LLM 上下文窗口的配合

### 6.1 RAG 全流程中的两个窗口

```
用户提问
    ↓
[Embedding 模型] ← 上下文窗口：问题向量化
    ↓
向量检索
    ↓
[LLM] ← 上下文窗口：问题 + 检索到的文档 + 对话历史
    ↓
生成回答
```

### 6.2 两个窗口的关系

| 维度 | Embedding 窗口 | LLM 窗口 |
|------|---------------|---------|
| **作用对象** | 单个文档片段 | 整个对话上下文 |
| **影响** | 分块大小上限 | 能同时参考的信息量 |
| **典型大小** | 2048 - 8192 tokens | 8K - 128K tokens |
| **成本敏感度** | 低（一次文档处理）| 高（每次问答都消耗）|

### 6.3 实际例子

假设：
- Embedding 窗口：2048 tokens
- LLM 窗口：8192 tokens
- 分块大小：500 字符（≈ 300 tokens）
- 检索 top_k：5

**LLM 一次看到的内容：**
```
系统提示词（≈ 100 tokens）
+ 对话历史（≈ 500 tokens）
+ 用户问题（≈ 50 tokens）
+ 5 个检索到的 chunk（5 × 300 = 1500 tokens）
─────────────────────────────────────
总计：≈ 2150 tokens（远小于 8192 上限）
```

## 七、常见问题解答

### Q1: 超过上下文窗口的文本会怎样？

**A:** 通常有两种处理方式：
1. **截断**：只保留前 N 个 token，后面的直接丢弃
2. **报错**：模型拒绝处理，返回错误信息

**后果：** 语义不完整，向量质量下降，检索失败。

### Q2: 为什么不都用最大窗口的模型？

**A:** 权衡考虑：
- **成本**：大窗口模型更贵
- **性能**：大窗口处理速度更慢
- **需求**：大多数场景不需要超大窗口
- **质量**：文本过长会稀释语义

### Q3: 如何选择合适的分块大小？

**A:** 参考以下步骤：
1. 查看 Embedding 模型的最大窗口
2. 乘以安全系数（0.7-0.8）
3. 根据文档特点调整（代码小一些，散文大一些）
4. 做 A/B 测试，评估检索质量

### Q4: Token 和字符怎么换算？

**A:** 经验公式：
- 中文：token ≈ 字符数 ÷ 1.5
- 英文：token ≈ 字符数 ÷ 4
- 混合：按比例估算

**更准确的方式：** 使用对应的 tokenizer 实际计算。

## 九、总结

### 9.1 核心要点

1. **上下文窗口** = 模型一次能处理的最大文本长度
2. **Embedding 窗口** 决定了文档分块的上限
3. **安全系数** 很重要，不要卡着上限用
4. **Token ≠ 字符**，需要按语言换算

### 9.2 快速参考表

| 场景 | 推荐分块大小 | 适用窗口 |
|------|-------------|---------|
| FAQ/问答对 | 100-300 字符 | 2048+ |
| 技术文档 | 300-600 字符 | 2048+ |
| 小说/散文 | 500-1000 字符 | 8192+ |
| 代码片段 | 200-400 字符 | 2048+ |

### 9.3 最佳实践

✅ **做**：
- 先确认 Embedding 模型的窗口大小
- 分块时留 20-30% 的安全余量
- 根据文档类型调整分块策略
- 测试不同分块大小的检索效果

❌ **不要**：
- 分块大小等于窗口上限
- 忽略 Token 和字符的区别
- 所有文档用同样的分块大小
- 忘记考虑重叠分块
