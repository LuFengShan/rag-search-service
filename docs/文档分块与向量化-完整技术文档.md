# 文档分块与向量化 — 完整技术文档

## 1. 概述

本系统对上传到知识库的文档执行 **"解析 → 分块 → 向量化 → 入库"** 全流程自动化处理。整个流程在独立线程池中异步执行，不阻塞用户的 HTTP 请求。

**核心代码涉及的文件：**

| 文件 | 职责 |
|------|------|
| `service/DocumentService.java` | 上传入口：保存文件、插入记录、触发异步处理 |
| `service/DocumentProcessService.java` | ⭐ 异步处理主流程：解析 → 分块 → 向量化 |
| `service/DocumentParserService.java` | 文档解析：Tika / 纯文本 / ZIP 回退 |
| `service/VectorService.java` | 向量生成（Embedding） + 相似度检索 |
| `config/DashScopeEmbeddingConfig.java` | 阿里云 DashScope Embedding 模型配置 |
| `document/MarkdownFrontmatterExtractor.java` | Markdown frontmatter 解析器 |
| `mapper/DocumentChunkVectorMapper.java` | 向量数据入库 SQL |

**涉及的数据表：**

| 表 | 说明 |
|---|------|
| `document_chunk` | 文档分块记录（id, document_id, chunk_index, content, embedding, created_at） |
| `documents` | 文档主表（status 字段驱动状态机） |

## 3. 阶段 2：文档内容解析

> 代码：[DocumentParserService.java](file:///Users/sunguangxu/Documents/trae_projects/langchaindemo/rag-search-service/src/main/java/com/example/rag/service/DocumentParserService.java)

解析策略分三层：

| 优先级 | 条件 | 方式 |
|--------|------|------|
| 1 | `.txt`, `.md`, `.json` 等 | 直接读取文件为 UTF-8 字符串 |
| 2 | PDF, Word, PPT 等 | Apache Tika `AutoDetectParser` 自动检测并提取文本 |
| 3 | Tika 失败 + ZIP 格式（docx/xlsx） | ZIP 归档回退，提取其中 HTML/XML/TXT 并剥离标签 |

输出：**纯文本字符串**，已清理零宽字符和控制字符。

## 4. 阶段 3：智能分块（Chunking）⭐核心

> 代码：[DocumentProcessService.java](file:///Users/sunguangxu/Documents/trae_projects/langchaindemo/rag-search-service/src/main/java/com/example/rag/service/DocumentProcessService.java)

系统根据文档类型采用不同的分块策略：

```
isMarkdown(document)?
├── YES → chunkMarkdownDocument(content)
│         ├── 有 frontmatter → chunkCarMarkdown(content)   ← 车系语义分块
│         └── 无 frontmatter → chunkGenericDocument(body)   ← 段落级通用分块
└── NO  → chunkGenericDocument(content)                    ← 段落级通用分块
```

### 4.1 车系文档语义分块（chunkCarMarkdown）

这是本系统的核心分块策略，专门针对带 frontmatter 元数据的车系 Markdown 文档。

#### 4.1.1 输入示例（丰田凯美瑞文档）

```
---
brand: 丰田
model: 凯美瑞
priceRange: 17.98-26.98万
carType: 中型轿车
fuelType: 燃油/双擎混动
tags: [商务家用, 舒适大空间, 混动省油, B级标杆, 保值率高]
competitors: [本田雅阁, 大众帕萨特, 日产天籁, 比亚迪汉]
targetUsers: 商务人士/中产家庭
salesPoints: 丰田双擎/舒适标杆/B级销冠/保值率高/宜商宜家
---

## 基本信息
丰田凯美瑞2025款基于TNGA-K平台...

## 配置与价格
| 配置版本 | 指导价 | 动力 |
|----------|--------|------|
| 2.0E 精英版 | 17.98万 | 2.0L+CVT |
...

## 动力与油耗
2.0L版最大功率131kW...

## 核心卖点
1. **混动标杆**：...
...

## 适合人群
- 预算18-25万的商务人士
...

## 竞品对比
vs 本田雅阁：...
```

#### 4.1.2 处理流程

**步骤 1：提取元数据**

使用 [MarkdownFrontmatterExtractor](file:///Users/sunguangxu/Documents/trae_projects/langchaindemo/rag-search-service/src/main/java/com/example/rag/document/MarkdownFrontmatterExtractor.java) 解析 `---` 包裹的 YAML frontmatter：

```
正则：^---\s*\n(.*?)\n---\s*\n(.*)   →   组1=YAML块   组2=正文
```

解析 YAML 支持：
- **简单键值对**：`brand: 丰田`
- **引号值**：`year: "2025"` → 自动去引号
- **列表**：`tags:\n  - 商务家用\n  - 混动省油` → `List<String>`
- **空列表**：`tags: []` → 跳过

输出 `CarDocumentMetadata` 对象：

```java
CarDocumentMetadata {
    brand = "丰田",
    model = "凯美瑞",
    priceRange = "17.98-26.98万",
    carType = "中型轿车",
    fuelType = "燃油/双擎混动",
    tags = ["商务家用", "舒适大空间", "混动省油", "B级标杆", "保值率高"],
    competitors = ["本田雅阁", "大众帕萨特", "日产天籁", "比亚迪汉"],
    targetUsers = "商务人士/中产家庭",
    salesPoints = "丰田双擎/舒适标杆/B级销冠/保值率高/宜商宜家"
}
```

**步骤 2：构建元数据前缀**

将元数据格式化为可嵌入 chunk 的文本前缀：

```
【品牌:丰田/凯美瑞】
【车型:中型轿车】【能源:燃油/双擎混动】
【价格:17.98-26.98万】
【标签:商务家用、舒适大空间、混动省油、B级标杆、保值率高】
【竞品:本田雅阁、大众帕萨特、日产天籁、比亚迪汉】
【卖点:丰田双擎、舒适标杆、B级销冠、保值率高、宜商宜家】
【适合人群:商务人士/中产家庭】
```

> 📌 **设计意图**：这些元数据随每个 chunk 一起 embedding，检索时用户查询"省油的 B 级车"，向量相似度自然会匹配到 `标签:混动省油` 和 `车型:中型轿车`，无需额外的过滤步骤。

**步骤 3：按二级标题拆分正文**

正则 `(?=\n##\s)` 在 `换行 + ## + 空格` **之前**的位置切开，切开后每个章节保留完整的 `##` 标题行：

```java
String[] sections = body.split("(?=\\n##\\s)");
// 产生 6 个章节：
// [0] "## 基本信息\n丰田凯美瑞2025款..."
// [1] "## 配置与价格\n| 配置版本 | ..."
// [2] "## 动力与油耗\n2.0L版最大功率..."
// [3] "## 核心卖点\n1. **混动标杆**..."
// [4] "## 适合人群\n- 预算18-25万..."
// [5] "## 竞品对比\nvs 本田雅阁..."
```

> 📌 **正则说明**：`(?=...)` 是正向预查，匹配位置但不消耗字符，因此 `##` 标题行保留在每个切片的开头。

**步骤 4：组装最终 chunk**

每个章节与元数据前缀拼接，形成自包含的语义单元：

```
Chunk 1（127 chars）：
【品牌:丰田/凯美瑞】
【车型:中型轿车】【能源:燃油/双擎混动】
【价格:17.98-26.98万】
【标签:商务家用、舒适大空间、混动省油...】
...

## 基本信息
丰田凯美瑞2025款基于TNGA-K平台，车身尺寸：4885×1840×1455mm，轴距2825mm...

─────────────────────

Chunk 2（263 chars）：
【品牌:丰田/凯美瑞】
...（相同元数据）...

## 配置与价格
| 配置版本 | 指导价 | 动力 |
|----------|--------|------|
| 2.0E 精英版 | 17.98万 | 2.0L+CVT |
...

─────────────────────

...（共 6 个 chunk）
```

#### 4.1.3 最终输出规格

```
Chunked car doc [丰田 凯美瑞] into 6 chunks (max 6000 chars/chunk)
  chunk 0: ## 基本信息 (127 chars)
  chunk 1: ## 配置与价格 (263 chars) ← 含表格
  chunk 2: ## 动力与油耗 (118 chars)
  chunk 3: ## 核心卖点 (192 chars)   ← 含列表
  chunk 4: ## 适合人群 (121 chars)
  chunk 5: ## 竞品对比 (139 chars)
```

#### 4.1.4 策略优势

| 优点 | 检索效果 |
|------|---------|
| 语义完整 | 问"凯美瑞价格"→ 命中 Chunk 2（含完整价格表），而非碎片化的 `17.98万` 孤立文本 |
| 元数据锚定 | 每个 chunk 自带品牌/车系/标签，检索时自然匹配到关键词 |
| 长度适中 | 120~260 字符 ≈ 60~130 tokens，embedding 质量高 |
| 表格保留 | 表格随所在章节完整保留，不被 `\n` 切碎 |

### 4.2 超长章节二次拆分（splitLongChapter）

当单个章节 > 6000 字符时触发，策略：

```
┌─ 有三级标题（###）？
│   YES → 按 ### 拆分为子章节
│         每个子章节独立拼元数据前缀
│         子章节仍超长 → 递归按段落拆
│
└─ 无三级标题？
    → 按段落边界拆分（双空行 \n\n）
      每个子段落拼元数据前缀
```

**段落边界拆分示意：**

```
原章节（8000 chars）：
  "## 竞品对比\n\nvs 本田雅阁：...\n\nvs 比亚迪汉：...\n\nvs 大众帕萨特：..."

拆分后：
  Sub-chunk A:
    【元数据...】
    ## 竞品对比
    vs 本田雅阁：...（3000 chars）

  Sub-chunk B:
    【元数据...】
    ## 竞品对比
    vs 比亚迪汉：...（2500 chars）

  Sub-chunk C:
    【元数据...】
    ## 竞品对比
    vs 大众帕萨特：...（2500 chars）
```

> 每个子 chunk 都重新拼接了完整元数据前缀 + 章节标题，确保每个 chunk 都自包含。

### 4.3 通用文档分块（chunkGenericDocument）

非车系 MD 文档和 PDF/Word 等通用文档使用此策略：

```
按双空行（\n\n）切分为段落
  →
按 6000 字符上限累加段落，超长即保存当前 chunk
```

简单高效，适合无结构标记的文档。

### 4.4 分块关键参数

| 参数 | 值 | 依据 |
|------|-----|------|
| `MAX_CHUNK_CHARS` | 6000 | ≈ 1500 tokens（中文），适配 text-embedding-v4 上下文窗口 |
| 章节分割标记 | `## `（二级标题） | 对应文档的语义单元边界 |
| 元数据格式 | `【标签:值】` | 中文全角括号在 embedding 中自然参与语义匹配 |

## 5. 阶段 4：向量化（Embedding）

> 代码：[VectorService.java](file:///Users/sunguangxu/Documents/trae_projects/langchaindemo/rag-search-service/src/main/java/com/example/rag/service/VectorService.java) + [DashScopeEmbeddingConfig.java](file:///Users/sunguangxu/Documents/trae_projects/langchaindemo/rag-search-service/src/main/java/com/example/rag/config/DashScopeEmbeddingConfig.java)

### 5.1 调用链路

```
VectorService.embed(text)
  │
  ├── 第 1 层重试：@Retryable（Spring Retry）
  │   最大 3 次，指数退避 1s→2s→4s
  │
  └── embeddingModel.embed(text)
        │
        └── DashScopeEmbeddingModel.doEmbedWithRetry(text)
              │
              ├── 第 2 层重试：内嵌循环
              │   最大 3 次，指数退避 1s→2s→4s
              │
              └── callEmbeddingApi(text)
                    │
                    POST https://dashscope.aliyuncs.com/compatible-mode/v1/embeddings
                    Body: {"model":"text-embedding-v4","input":"...","dimensions":1536,
                           "encoding_format":"float"}
                    │
                    返回：{ "data": [{ "embedding": [0.123, -0.456, ...] }] }
```

> 📌 **双层重试设计**：DashScopeEmbeddingModel 内层处理单次 HTTP 调用级重试（网络抖动等）；VectorService 外层的 `@Retryable` 处理整体嵌入失败（如返回空响应），失败后走 `@Recover` 降级方法。

### 5.2 API 端点详情

| 项目 | 值 |
|------|-----|
| **基础 URL** | `https://dashscope.aliyuncs.com/compatible-mode/v1` |
| **端点** | `POST /embeddings` |
| **模型** | `text-embedding-v4` |
| **输出维度** | 1536 |
| **编码格式** | `float`（浮点数组） |
| **认证** | `Authorization: Bearer {DASHSCOPE_API_KEY}` |

### 5.3 请求体示例

```json
{
  "model": "text-embedding-v4",
  "input": "【品牌:丰田/凯美瑞】... ## 配置与价格\n| 配置版本 | 指导价 |...",
  "dimensions": 1536,
  "encoding_format": "float"
}
```

### 5.4 响应 → float[] 转换

DashScope 返回 `List<Double>`，逐元素转为 `float[]`：

```java
// API 响应 DTO 映射
{
  "data": [{
    "embedding": [0.0067, -0.0377, 0.0377, ...],  // List<Double>, 1536 个元素
    "index": 0
  }]
}

// Java 转换
float[] result = new float[1536];
for (int i = 0; i < 1536; i++) {
    result[i] = embedding.get(i).floatValue();
}
```

### 5.5 入库 SQL

```sql
INSERT INTO document_chunk (
    id, document_id, chunk_index, content, embedding, created_at
) VALUES (
    #{id}, #{documentId}, #{chunkIndex}, #{content},
    CAST(#{embedding} AS vector),  -- 字符串 → pgvector 类型
    NOW()
)
```

`embedding` 字段的字符串格式为 `[0.0067, -0.0377, 0.0377, ...]`，PostgreSQL 的 `CAST(... AS vector)` 将其转换为 pgvector 内部存储格式。

### 5.6 降级策略

```
DashScope API 可用？
├── YES → 150-D 真实向量（text-embedding-v4）
└── NO  → 基于文本 hash 的伪随机向量
          embedFallback(text):
            hash = text.hashCode()
            for i in 0..1535:
              embedding[i] = sin(i * hash * 0.01) * 0.5 + 0.5
```

> ⚠️ 降级向量仅保证同文本产生相同向量（可检索到自身），但不同文本间的语义相似度不可靠。

## 6. 检索时的向量匹配

> 代码：[VectorService.java#searchByCosineSimilarity](file:///Users/sunguangxu/Documents/trae_projects/langchaindemo/rag-search-service/src/main/java/com/example/rag/service/VectorService.java#L96)

当用户提问时，问题文本同样经过 embed → 在 `document_chunk` 表做向量相似度检索 → 取 top-K 最相似的 chunk。

### 6.1 三种相似度算法

| 算法 | pgvector 运算符 | 公式 | 特点 |
|------|:---:|------|------|
| **余弦相似度**（默认） | `<=>` | `cos(θ) = A·B / (‖A‖·‖B‖)` | 范围 [-1,1]，值越大越相似。最常用 |
| L2 距离 | `<#>` | `√Σ(Ai-Bi)²` | 欧氏距离，值越小越相似 |
| 内积 | `<@>` | `Σ(Ai·Bi)` | 值越大越相似 |

### 6.2 检索示例

```
用户问题："凯美瑞混动版落地多少钱"

步骤 1：embed("凯美瑞混动版落地多少钱") → float[1536]

步骤 2：SELECT content FROM document_chunk
        WHERE embedding <=> CAST('[0.123, -0.456, ...]' AS vector)
        ORDER BY embedding <=> CAST('...' AS vector)
        LIMIT 5;

步骤 3：返回 Top-5 chunk：
  Chunk 2: "【品牌:丰田/凯美瑞】...## 配置与价格\n| 2.5HG 豪华版 | 21.98万 | 2.5L混动 |"
  Chunk 3: "【品牌:丰田/凯美瑞】...## 动力与油耗\n2.5L双擎版系统综合功率160kW..."
  Chunk 4: "【品牌:丰田/凯美瑞】...## 核心卖点\n1. **混动标杆**..."

步骤 4：Top-5 chunk 拼接为上下文 → 送给 LLM 生成最终答案
```

> 📌 注意 Chunk 2 排名最高——因为"混动版"匹配了元数据中的 `能源:燃油/双擎混动`，"落地多少钱"匹配了配置表中的价格信息。**元数据 embedding 起到了隐式过滤和语义对齐的作用。**

## 7. 分块效果对比

### 旧策略（500字符硬限制 + frontmatter独立chunk）

```
凯美瑞.md → 约 8 个 chunk：

Chunk 0: "---\nbrand: 丰田\nmodel: 凯美瑞\n..."  ← frontmatter 孤立
Chunk 1: "## 基本信息\n丰田凯美瑞2025款基于TNGA-K平台..."  ← 无元数据
Chunk 2: "| 配置版本 | 指导价 | 动力 |\n|----------|--------|----|"
Chunk 3: "| 2.0E 精英版 | 17.98万 | 2.0L+CVT |\n| 2.0G 豪华版 |..."
Chunk 4: "| 2.5HG 豪华版 | 21.98万 | 2.5L混动 |\n| 2.5HS 锋尚版 |..."
Chunk 5: "| 2.5HQ 旗舰版 | 26.98万 | 2.5L混动 |"  ← 价格表被切成碎片
...
```

❌ **问题**：frontmatter 孤立无检索价值；价格表被切成无上下文的碎片；chunk 不包含元数据。

### 新策略（元数据前缀 + ## 语义分块）

```
凯美瑞.md → 6 个 chunk：

Chunk 0: 元数据 + "## 基本信息" + 尺寸/轴距 (127 chars)
Chunk 1: 元数据 + "## 配置与价格" + 完整价格表 (263 chars)  ← 表格完整保留
Chunk 2: 元数据 + "## 动力与油耗" + 油耗数据 (118 chars)
Chunk 3: 元数据 + "## 核心卖点" + 卖点列表 (192 chars)
Chunk 4: 元数据 + "## 适合人群" + 用户画像 (121 chars)
Chunk 5: 元数据 + "## 竞品对比" + 对比分析 (139 chars)
```

✅ **改进**：每个 chunk 都是自包含的语义单元；表格/列表完整保留；元数据嵌入提升检索精准度。

```yaml
        options:
          model: deepseek-chat
      embedding:
        enabled: false                  # ⚠️ 禁用 OpenAI Embedding，使用 DashScope
```

> ⚠️ 必须设置 `spring.ai.openai.embedding.enabled=false`，否则 Spring AI 默认的 OpenAI Embedding 会与 DashScope 冲突。

## 9. 故障排查

| 现象 | 可能原因 | 检查方法 |
|------|---------|---------|
| chunk 数量为 0 | 文档解析返回空 | 查看日志 `"no extractable content"` |
| 向量 API 持续失败 | DASHSCOPE_API_KEY 未设置或无效 | `curl https://dashscope.aliyuncs.com/compatible-mode/v1/embeddings ...` |
| chunk 内容不含元数据 | 非车系 MD 文档（无 frontmatter） | 检查文档前三行是否为 `---` |
| 价格表被拆分 | 章节超 6000 字符触发了二次拆分 | 检查日志中的章节字符数 |
